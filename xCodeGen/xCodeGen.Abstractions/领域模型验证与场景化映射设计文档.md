# 领域模型验证与场景化映射技术设计文档（V1.25）

本设计以 **“标注驱动生成 + 手写自治”** 为核心，xCodeGen 负责生成 80% 以上的场景化映射与基础验证代码，剩余复杂业务逻辑由开发者在 `.Logic.cs` 中手写扩展。

## 1. 核心设计原则（Foundational Principles）

为了平衡性能、安全性与开发效率，本系统遵循以下三大基础策略：

- **策略一：持久化信任（Persistence Trust）**：
  
  来自数据库（持久化层）的数据在 DTO 层面默认信任其已满足物理约束。在常规业务场景下，跳过重复的长度、格式等物理校验。
  
  - `IsFromPersistentSource == true` 时 DTO 跳过校验。

- **策略二：回填驱动验证（Mapping-Driven Validation）**：
  
  只有当前场景下允许修改（`CanModify`）且需要回填的字段，才执行该字段的验证。**不修改 → 不回填 → 不验证**。

- **策略三：模型终验制（Model Final Validation）**：
  
  Model 层的 `Create/Update` 动作一律显式调用 `Validate()`。即使 DTO 层面因只读跳过了某些字段，Model 依然负责所有数据（包括系统填充字段）的最终一致性守卫。

注意：以 DataAnnotations 为基础生成验证规则，后续根据需要兼容 FluentValidation。

## 2. 属性行为决策矩阵（Property Decision Matrix）

行为由元数据（`Column`、`DtoField`）与场景（`EnumSceneFlags`）通过 `CodeGenPolicy` 决策引擎共同决定：

| **模式/特性组合**                 | **DTO 定义** | **ApplyToEntity（回填）** | **ValidateData（验证）** | **FromEntity（脱敏）**            | **典型例子**      |
| --------------------------- | ---------- | --------------------- | -------------------- | ----------------------------- | ------------- |
| **`Ignore`**                | 否          | 否                     | 否                    | -                             | 内部敏感字段        |
| **`CanModify = false`**     | 是          | 否                     | 否                    | 正常                            | `UId`, `Code` |
| **`UpdateReadOnly = true`** | 是          | 仅 `Create` 场景可写       | 仅 `Create` 场景校验      | 正常                            | 证件号、初始类别      |
| **`Masking = true`**        | 是          | 取决于 `CanModify`       | 取决于 `CanModify`      | 根据 MaskPattern<br/>**执行掩码替换** | 手机号、邮箱        |

注：`DtoFieldAttribute` 仅定义字段与场景的关系、是否自动生成查询方法（目前仅支持`IsUnique`）。

## 3. 自动生成查询方法（Query Method Automation）

本章节定义了 `xCodeGen` 如何根据模型元数据（Attributes）自动在 `Service.generated.cs` 中构建高可用、高性能、强扩展性的数据访问接口逻辑标准。

### 3.1 自动生成规则与优先级（Generation Rules & Priorities）

查询方法的生成采用“冲突回避”机制，按照以下优先级顺序扫描元数据。一旦某组字段已被高优先级规则处理，低优先级规则将自动跳过该组合，以确保接口的唯一性与整洁性。

| **优先级**    | **触发源**                                                        | **方法前缀**                                          | **返回类型**                                                                    | **说明**                                                                     |
| ---------- | -------------------------------------------------------------- | ------------------------------------------------- | --------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| **P1（最高）** | `Column(IsPrimary = true)`                                     | `GetBy`                                           | `Task<TDto>`                                                                | 主键查询。作为底层最核心的单记录查询入口。                                                      |
| **P2**     | `DtoField(SearchGroup="...")`<br>`DtoField(IsSearchable=true)` | `GetBy` / `SelectBy` / `CountBy` / `SelectPageBy` | `Task<TDto>` / `Task<List<TDto>>` / `Task<long>` / `Task<PageResult<TDto>>` | **语义化与动态搜索**。基于业务定义的查询组或单一搜索字段，自动生成支持智能解析的四位一体接口。此级别优先级最高，自动覆盖物理索引的同名字段组合。 |
| **P3**     | `Index(IsUnique = true)`                                       | `GetBy`                                           | `Task<TDto>`                                                                | 唯一索引查询。自动根据索引包含的字段生成严格匹配的单记录获取接口。                                          |
| **P4（最低)** | `Index(IsUnique = false)`                                      | `SelectBy` / `CountBy` / `SelectPageBy`           | `Task<List<TDto>>` / `Task<long>` / `Task<PageResult<TDto>>`                | 普通索引查询。生成返回集合及分页包装对象的动态列表查询接口。                                             |

### 3.2 命名算法逻辑（Naming Convention）

为了使生成的代码具备极高的可读性，消除冗长的表名污染，方法后缀采用以下智能探测与剥离逻辑：

1. **单字段优先**：如果是单一属性的索引或搜索字段，直接使用属性名（如：`Status` -> `SelectByStatusAsync`）。

2. **SearchGroup 优先**：若定义了 `SearchGroup`，直接使用该名称作为后缀（如：`SearchGroup="Mobile"` -> `SelectByMobileAsync`）。

3. **复合索引名剥离**：若为 `IndexAttribute` 且为多字段复合，程序将解析索引名称（匹配 `idx_...`），并**智能剥离当前实体类名**。
   
   - *示例*：模型 `MerchantInfo` 中定义了索引 `idx_merchant_info_credit_code`。算法提取 `MerchantInfoCreditCode` 后，识别并剔除类名 `MerchantInfo`，最终生成 `GetByCreditCodeAsync`。

4. **保底拼接**：若上述信息均缺失，则按参数顺序拼接字段名（如：`GetByProvinceAndCityAsync`）。

### 3.3 统一查询与统计体系（Unified Query System）

自动化查询方法摒弃了简单的 SQL 封装，提供了一套高度工程化、对调用方极其友好的动态查询体系：

1. **SearchGroup 与 IsSearchable 的彻底解耦**：
   
   - **`IsSearchable = true`** 的核心职责：触发生成该字段**独立的四位一体查询组**，并**全局开启**针对该字段的模糊/范围匹配逻辑。
   
   - **`SearchGroup = "X"`** 的核心职责：将字段加入**复合查询组 "X"** 中。
   
   - **组合规则**：若一个字段同时指定了两者，它既会生成独立的单字段查询，也会参与复合查询。若一个字段在 `SearchGroup` 中但未指定 `IsSearchable`，则在复合查询中该字段将严格采用精确匹配（`==`）。

2. **智能字符串解析约定（Smart String Parsing）**：
   
   对于所有非唯一查询（`SelectBy/SelectPageBy/CountBy`），只要参数类型为 `string`，系统会自动挂载基于 `DomainServiceBase.ParseSearchValue` 的智能解析器，将匹配模式的选择权完全交由调用方（如前端的搜索框约定）：
   
   - **默认行为**：精确匹配（例如传入 `"VIP"`，底层生成 `== "VIP"`）。
   
   - **`*` 开头**：触发模糊搜索（例如传入 `"*VIP"`，底层生成 `.Contains("VIP")`）。
   
   - **`**` 开头**：转义匹配（例如传入 `"**VIP"`，底层恢复为对真实数据中带星号的精确匹配 `== "*VIP"`）。

3. **范围与可空动态匹配**：
   
   - **日期类型**：若开启 `IsSearchable`，自动将单参数裂变为 `beginDate` 和 `endDate`，并生成 `>=` 与 `<=` 范围判断；否则生成可空的精确相等匹配。
   
   - **其余非唯一参数**：统一转化为可空类型（`?`），底层自动挂载 `if(val.HasValue)`，实现全量与带条件的无缝切换。

4. **四位一体与双分页兼容**：
   
   对于所有非唯一组合，自动成套生成共享一致参数签名的接口：
   
   - **SelectPageBy 模式**：基于 `PageNumber`/`PageSize`，返回带有 `TotalCount` 和 `TotalPages` 的 `PageResult<TDto>` 对象，适配 MIS 系统与中后台表格。
   
   - **SelectBy 模式**：基于 `Skip`/`Limit`，返回 `List<TDto>`，适配移动端 App/小程序的瀑布流。
   
   - **CountBy 模式**：单独获取该条件组合下的总记录数。

### 3.4 内部与外部接口隔离（Internal vs Public API）

架构严格划分表现层与底层引擎：

- **公共接口（Public DTO）**：所有对外暴露的查询方法必须返回 `Dto` 或 `PageResult<Dto>`，避免持久化实体被 WebAPI 等高层直接序列化或暴漏。

- **内部接口（Internal Entity）**：底层原子方法（如 `InternalGetAsync`、`InternalSelectAsync`）修饰符为 `internal`，返回 `Model` 实体，供同程序集内跨 Service 的领域服务编排使用。

- **安全护城河**：所有内部集合查询强制注入 `Math.Clamp` 与 `Math.Max` 边界检查（如限制 `limit` 最大为 1000），从源头阻断恶意参数导致的数据库 OOM。

### 3.5 写操作封装与只读模型（CUD & Read-Only）

Service 自动生成基于 DTO 的高层业务写接口，实现样板代码零编写：

- **只读视图支持（`IsView`）**：若通过元数据 `GenerateCode(Type = "View")` 或配置字典标记为视图，系统将自动屏蔽生成所有 `Create`、`Update`、`Delete` 方法，仅保留安全只读查询接口。

- **创建封装（CreateAsync）**：执行 `dto.ValidateData` -> `dto.ApplyToEntity` -> `InternalCreate`，并返回包含数据库自增值（如 ID、CreateTime）的全新 `DTO`。

- **更新封装（UpdateAsync）**：自动合并库内旧数据与 DTO 新值，通过 `scene` 标记防范只读字段被非法篡改。

- **智能删除（DeleteAsync）**：内部自动探测模型是否包含 `IsDeleted` 属性，智能路由到 `InternalSoftDelete`（软删）或 `InternalHardDelete`（硬删），并触发完整的 `OnBeforeDelete` / `OnAfterDelete` 生命周期钩子。

### 3.6 接口能力矩阵示例（Capabilities Matrix）

针对 `MerchantInfo` 模型，生成的 Public 方法矩阵如下：

| **方法名称**                | **参数结构**                              | **说明**                                                    |
| ----------------------- | ------------------------------------- | --------------------------------------------------------- |
| `GetByIdAsync`          | `long id`                             | 强主键获取，返回单条 `MerchantInfoDto`。                             |
| `SelectPageAsync`       | `predicate`, `pageNumber`, `pageSize` | 默认标准分页查询，内置 `QueryForUser` 租户过滤，返回 `PageResult`。          |
| `GetByCreditCodeAsync`  | `string creditcode`                   | 唯一键获取，基于索引自动剥离了冗余类名。                                      |
| `SelectPageByNameAsync` | `string name` （支持智能解析）                | 独立生成的单字段查询，支持传入 `*XXX` 进行模糊匹配。                            |
| `SelectByKeywordAsync`  | `string code`, `string name`...       | 基于 `SearchGroup="Keyword"` 生成的组合查询。                       |
| `CountByKeywordAsync`   | 同上                                    | 自动匹配上述 SearchGroup 条件的总数查询。                               |
| `DeleteAsync`           | `long id`                             | 统一删除入口，底层根据模型结构自动路由软/硬删除。                                 |
| `GetGraphQLQueryable`   | 无参数                                   | **特例接口**：返回 `IQueryable<Model>` 供 GraphQL 引擎进行数据库层面的按需投影。 |

### 3.7 核心决策引擎：CodeGenPolicy

该静态类已编译入 `xCodeGen.Core`，用于统一模板生成期与运行期的判定逻辑：

- **`GetSearchGroups(classMeta)`**：核心探针引擎。负责扫描 `DtoField` 和 `IndexAttribute`，合并同名业务组，标记 `IsUnique` 状态，并执行类名剥离 (`ToPascalCase`） 及对物理索引的智能去重操作。

- **`IsSearchable(PropertyMetadata)`**：字段级探针。为 Razor 模板提供该字段是否开启高级搜索模式的判定依据。

- **`ShouldMapToEntity / ShouldValidateDto`**：判断当前场景是否需要执行属性赋值及 DTO 校验。

## 4. 掩码系统规格 （Masking System Specification）

引入动态掩码逻辑，由 `MaskHelper` 静态类实现：

- **模式字符 （Pattern Characters）**：
  
  - `?`：匹配并保留真实值的**单个**字符。
  
  - `?*`：贪婪匹配并保留真实值的**连续多个**字符。
  
  - `#`：匹配真实值的**单个**字符并替换为遮蔽符 `*`。
  
  - `#*`：匹配真实值的**多个**字符并替换为遮蔽符 `*`。
  
  - **转义**：使用 `\` 处理原义字符，如 `\#`、`\*`。

- **示例**：
  
  - 手机号：`???####????` 或 `???#*????` $\rightarrow$ `139****5678`。
  
  - 邮件：`?#*@?#*.?#*` $\rightarrow$ `j***@g***.c*`。

- **掩码写回防护**：在 `ApplyToEntity` 时，若输入值等于该字段生成的掩码字符串（判定用户未编辑），**自动跳过回填**，防止掩码字符被写入数据库。

## 5. 自动化服务层逻辑 （Service Generation Specs）

服务层根据模型属性自动触发特定业务方法的生成：

### 5.1 状态管理逻辑 （Status Management）

系统通过识别布尔类型字段生成统一的状态切换接口，避免生成成对的冗余方法：

- **`IsEnabled` 识别**：生成 `InternalSetEnabledAsync(pk, bool enabled)`。

- **`IsDisabled` 识别**：生成 `InternalSetDisabledAsync(pk, bool disabled)`。

- **设计规范**：业务调用方通过 `bool` 参数控制目标状态，而非调用不同的方法名。

### 5.2 软删除与硬删除 （Soft/Hard Delete）

- **逻辑删除**：若存在 `IsDeleted` 字段，系统生成 `InternalSoftDeleteAsync`。

- **自动过滤**：`QueryForUser()` 会自动附加 `q.Where(x => !x.IsDeleted)` 过滤条件。

- **物理删除**：始终提供 `InternalHardDeleteAsync` 或 `InternalDeleteAsync` 作为底层指令。

### 5.3 业务钩子 （Partial Hooks）

生成的 Service 包含完整的生命周期钩子，允许开发者在不修改生成代码的情况下干预业务流程：

- **查询过滤**：`OnQueryFiltering` （基础查询）、`OnGraphQLFiltering` （接口查询）。

- **持久化前后**：`OnBeforeCreate/OnAfterCreate`, `OnBeforeUpdate/OnAfterUpdate`, `OnBeforeDelete/OnAfterDelete`。

## 6. 验证流与异常处理

### 6.1 验证分层逻辑

1. **自动物理验证 （`ValidateDataCore`）**：利用共享子模板 `_InternalValidation.cshtml` 生成。错误消息通过 `Summary` 或 `DisplayName` 增强可读性，将“字段名”与“错误信息”解耦。

2. **手写业务验证 （`OnCustomValidate / OnBusinessValidate`）**：手写局部方法，处理跨字段、跨表的复杂业务规则。

### 6.2 结构化异常机制

- **聚合错误**：验证方法返回 `IEnumerable<ValidationResult>`。

- **`ValidationResultsException`**：自定义异常类，接收完整的错误集合。由全局过滤器拦截并返回结构化的前端反馈。

---

## 7. 特殊场景处理 （ForceValidate）

- **强制验证 （ForceValidate）**：在 `Model.Validate()` 层面，若场景为`ForceValidate`，**忽略 `IsFromPersistentSource` 标记**，强制重走全量物理校验，确保数据在规则变更后的合规性。

- **提权回填**：在特定系统场景下，通过决策引擎允许对部分只读字段进行回填。

---

## 8. 标准业务流示例

1. **DTO 校验**：`dto.ValidateData(scene)`（基于 `ShouldValidateDto`）。

2. **载入实体**：从数据库 `Get` 原始实体对象。

3. **场景映射**：`dto.ApplyToEntity(entity, scene)`（根据 `ShouldMapToEntity` 自动跳过 `UpdateReadOnly` 字段）。

4. **模型终验**：`entity.Validate(scene)`。

5. **持久化**：执行 `SaveChangesAsync()`。

## 9. 自动化服务层方法规格 （Service Generation Specs）

生成的领域服务（Service）旨在封装重复的持久化模板代码，通过受保护的（`protected`）指令式方法提供底层操作，同时通过公开的（`public`）查询接口对接表现层。

### 9.1 基础架构与依赖

- **命名空间**：自动对齐 `*.Services`。

- **继承关系**：继承自 `DomainServiceBase<TUser>`。

- **仓储访问**：自动注入 `IFreeSql` 并生成基于当前实体的 `IBaseRepository<TEntity>` 快捷属性 `Repo`。

### 9.2 查询方法规格 （R）

#### 1) 通用查询接口 （General Select）

- **`QueryForUser()` （Protected）**:
  
  - **核心逻辑**：基于 `_fsql.Select<T>()` 开启查询。
  
  - **安全策略**：若模型包含 `IsDeleted`，自动附加 `Where(x => !x.IsDeleted)` 过滤。
  
  - **扩展性**：调用 `OnQueryFiltering` 钩子。

- **`GetGraphQLQueryable()` （Public）**:
  
  - **核心逻辑**：基于 `QueryForUser()` 并调用 `OnGraphQLFiltering` 钩子。
  
  - **用途**：返回 `IQueryable<T>` 供表现层（GraphQL/OData）执行动态投影、分页与排序。

#### 2) 通用单一/谓词查询 （Predicate Based）

- **主键快捷查询 （PK Based）**:
  
  - **`TakeByIdAsync(pk)` （Public）**: 强制获取。若记录不存在，应抛出异常（典型用于业务必选实体加载）。
  
  - **`TakeOrDefaultByIdAsync(pk)` （Public）**: 安全获取。若记录不存在，返回 `null`。

- **业务支撑谓词查询 （Protected）**:
  
  - **`FirstAsync(predicate)` / `FirstOrDefaultAsync(predicate)`**: 单记录获取指令，支持排序。
  
  - **`SelectAsync(predicate)`**: 多记录获取指令，支持排序。
  
  - **`SelectPagedAsync(predicate, pageIndex, pageSize)`**: 分页指令，同时支持 `FreeSql` 的偏移分页与游标分页习惯。

#### 3) 自动化索引/分组查询 （Metadata Driven）

本部分由 `CodeGenPolicy` 决策并生成，遵循**“唯一键 GetBy，搜索组 First/Select”**的命名逻辑：

- **唯一约束查询 （Unique Result）**:
  
  - **来源**：`IndexAttribute.IsUnique == true` 或 `DtoFieldAttribute.IsUnique == true`。
  
  - **命名**：`GetBy[PropertyName]Async`。若是复合索引，则为 `GetBy[Prop1]And[Prop2]Async`。
  
  - **结果**：返回单条记录或 `null`。

- **SearchGroup 组合查询 （Multi/Single Result）**:
  
  - **来源**：`DtoFieldAttribute.SearchGroup` 分组。
  
  - **讨论方案**：使用 `SearchGroup` 显式声明来替代不确定的 `IndexAttribute.IsUnique == false` 自动生成。
  
  - **命名**：
    
    - `First[SearchGroup]Async` / `FirstOrDefault[SearchGroup]Async` （单记录版）。
    
    - `Select[SearchGroup]Async` （多记录版）。
    
    - `SelectPaged[SearchGroup]Async` （分页版）。
  
  - **示例**：若 `SearchGroup = "GradeAndClass"`，则生成 `FirstGradeAndClassAsync(string grade, string class, ...)`。

### 9.3 状态管理指令 （受保护）

- **`InternalSetEnabledAsync(pk, bool)`**：识别 `IsEnabled` 字段并生成状态切换指令。

- **`InternalSetDisabledAsync(pk, bool)`**：识别 `IsDisabled` 字段并生成状态切换指令。

### 9.4 持久化指令 （CUD - 受保护）

- **`InternalCreateAsync(entity)`**：封装 `OnBeforeCreate` -> `Insert` -> `OnAfterCreate`。

- **`InternalUpdateAsync(entity)`**：封装 `OnBeforeUpdate` -> `Update` -> `OnAfterUpdate`。

- **`InternalSoftDeleteAsync(pk)`**：仅在存在 `IsDeleted` 字段时生成，执行逻辑删除指令。

- **`InternalHardDeleteAsync(pk)`**：执行物理删除指令。
