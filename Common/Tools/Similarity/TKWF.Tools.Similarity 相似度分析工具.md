这份文档旨在为 `TKW.Framework.Common.Similarity` 模块提供全方位的技术说明。该模块作为框架的核心组件，旨在解决业务逻辑中常见的“文本去重”、“相似性评估”及“模糊匹配”需求。

# TKWF.Tools.Similarity 相似度分析工具

## 

## 1. 需求分析 (Demand Analysis)

在构建通用业务系统时，简单的字符串等值比较（`==` 或 `Equals`）往往无法满足以下复杂场景：

- **数据去重 (Deduplication)**：判断用户输入的名称（如“美团外卖”与“美团外卖 ”）是否指向同一实体。

- **纠错与建议 (Spell Checking)**：当搜索关键词与已有标签高度接近时，自动校正或给出提示。

- **语义近似 (Semantic Approximation)**：评估不同表述方式之间的内容重合度。

- **高性能要求**：在处理高并发请求或大规模基础数据时，相似度计算不能成为性能瓶颈，且需避免产生过多的 GC 压力。

---

## 2. 设计原理 (Design Principles)

### 2.1 架构设计

- **接口隔离**：定义 `ISimilarityChecker` 标准接口，实现“算法插件化”。

- **零/低分配 (Low Allocation)**：深度利用 .NET 10 的 `ReadOnlySpan<char>` 处理字符串，避免频繁创建 `string` 副本。

- **AOT 兼容**：代码不依赖任何动态代码生成（Reflection.Emit），完全支持 **Native AOT** 部署。

- **池化管理**：在大文本计算时，通过 `ArrayPool<int>` 复用中间计算数组，减少内存碎片。

### 2.2 核心算法选型

| **算法名称**                    | **核心原理**                     | **优势场景**               | **局限性**                  |
| --------------------------- | ---------------------------- | ---------------------- | ------------------------ |
| **Levenshtein (编辑距离)**      | 计算将 A 串转换成 B 串所需的最少编辑步数。     | **短文本**、拼写纠错、字符级精确对比。  | 对词序颠倒（如 A B vs B A）极其敏感。 |
| **Jaccard (Jaccard Index)** | 计算两个集合交集与并集的比例（此处使用 2-Gram）。 | **词序颠倒**、部分重合、关键词提取对比。 | 忽略了字符出现的顺序和频率。           |

这两种算法是文本处理领域的“绝代双骄”，它们从完全不同的维度审视文本的相似度。理解它们的本质差异，是构建智能系统的关键。

---

## 3. 算法核心原理

### 3.1 莱文斯坦距离 (Levenshtein Distance)

也称为**编辑距离**。它衡量的是将字符串 A 转换为字符串 B 所需的**最少单字符编辑操作次数**（插入、删除、替换）。

- **直观理解**：它像是一个“工匠”，在乎的是每一个字符的精确位置。

- **计算公式**：
  
  $$lev_{a,b}(i,j) = \begin{cases} \max(i,j) & \text{if } \min(i,j)=0 \\ \min \begin{cases} lev_{a,b}(i-1,j)+1 \\ lev_{a,b}(i,j-1)+1 \\ lev_{a,b}(i-1,j-1)+1_{(a_i \neq b_j)} \end{cases} & \text{otherwise} \end{cases}$$

### 3.2 Jaccard 相似度 (Jaccard Similarity)

它衡量的是两个**集合**的重叠程度，定义为交集大小除以并集大小。在文本处理中，通常先将文本拆分为 **N-Gram**（如 Bigram：相邻两个字组成的集合）。

- **直观理解**：它像是一个“会计”，只在乎两个篮子里的东西有多少是重合的，不在乎它们的摆放顺序。

- **计算公式**：
  
  $$J(A, B) = \frac{|A \cap B|}{|A \cup B|}$$

---

### 3.3 优缺点深度对比

| **特性**     | **Levenshtein (编辑距离)**           | **Jaccard (N-Gram 集合)**     |
| ---------- | -------------------------------- | --------------------------- |
| **关注点**    | **顺序**与**字符级差异**                 | **内容**与**重合度**              |
| **对顺序敏感度** | **极高**。词序颠倒会使距离剧增。               | **低**。非常适合词序变动的场景。          |
| **性能**     | $O(M \times N)$。虽有优化，但在长文本下开销较大。 | $O(N)$。构建集合并计算交集，速度极快。      |
| **容错性**    | 对“拼写错误”（加减错别字）有极佳的捕捉能力。          | 对“拼写错误”敏感度一般，取决于 N-Gram 大小。 |
| **长度敏感度**  | 长度差异越大，相似度越低。                    | 只要核心词重合，长短差异影响相对较小。         |

---

### 3.4 场景选择指南：什么时候选哪种？

#### 场景 A：选用 Levenshtein 的情况

> **核心诉求：精确、顺序敏感、处理拼写纠错。**

1. **用户注册/登录**：判断“ZhangSan”和“ZhanSan”是否是同一个人。

2. **验证码/短工单号**：处理由于 OCR 或人工输入造成的微小字符偏移。

3. **短文本标签清洗**：如“美团外卖” vs “美团外 买”（仅多一个空格或错一个字）。

4. **DNA 序列比对**：这种对碱基排列顺序有严格要求的生物信息学领域。

#### 场景 B：选用 Jaccard 的情况

> **核心诉求：抗乱序、大批量、关注关键词重合。**

1. **地址匹配**：如“北京市海淀区中关村”和“中关村（北京海淀区）”。由于 Jaccard 关注集合交集，它能一眼看出这俩是一回事。

2. **商品标题去重**：如“2026款 运动鞋 红色 42码” vs “红色 42码 运动鞋 2026款”。

3. **文章抄袭初步筛选**：快速判断两篇长文是否有大量重复的词汇集合。

4. **搜索引擎建议**：基于用户历史搜索词集，推荐内容高度重叠的相关搜索。

---

### 3.5 专家建议：混合策略

在实际的高级业务中，我们往往不只选一种。

- **漏斗式过滤**：先用 **Jaccard** 这种计算廉价的算法从海量数据中快速捞出潜在的候选集（召回阶段）。

- **精细化打分**：再对这些候选集使用 **Levenshtein** 进行精确的顺序比对（排序阶段）。

通过这种组合，你可以兼顾 Jaccard 的**速度/抗乱序**和 Levenshtein 的**精确度**。既然你在设计 `TKW.Framework`，建议在 API 中提供这两种算法的权重配置，让开发者能根据具体业务（是地址还是人名）自行调整比例。

## 4. 实现细节 (Implementation Details)

### 4.1 优化后的 Levenshtein 实现

传统的 Levenshtein 算法空间复杂度为 $O(M \times N)$。本模块采用了**滚动数组优化 (Iterative with two rows)**，并进一步缩减为**单行滚动**，空间复杂度降至 $O(min(M, N))$。

- **栈内存优化**：对于长度小于 256 的字符串，使用 `stackalloc` 分配内存，完全避开堆内存。

- **数组池支持**：针对超长文本，自动从 `ArrayPool` 借用空间。

### 4.2 字符级 Jaccard (Bigram) 实现

通过将字符串切分为长度为 2 的连续片段（Bigram），本模块可以有效处理诸如“北京烤鸭”与“烤鸭北京”这种词序变动但内容高度相似的情况。

## 5. 使用说明 (Usage Guide)

### 5.1 基础用法

模块设计为开箱即用，通过实例化具体的实现类即可：

```csharp
using TKW.Framework.Common.Similarity;

// 1. 初始化算法（建议在 DI 中注册为单例）
ISimilarityChecker checker = new LevenshteinSimilarity();

// 2. 直接计算评分
double score = checker.Calculate("瑞幸咖啡", "瑞星咖啡"); 
Console.WriteLine($"相似度: {score:P}"); // 输出: 相似度: 75.00%

// 3. 快速判断重复
bool isDup = checker.IsDuplicate("星巴克(中关村店)", "星巴克中关村", 0.7);
```

### 5.2 依赖注入 (Dependency Injection)

在 `IServiceCollection` 中注册：

```csharp
// 默认推荐使用编辑距离
services.AddSingleton<ISimilarityChecker, LevenshteinSimilarity>();
```

## 6. 后续扩展 (Future Expansion)

### 6.1 Plus 扩展包方向

对于更复杂的业务场景，建议通过独立扩展包引入以下功能：

- **SimHash (TKW.Extensions.Similarity.SimHash)**：
  
  - 针对**长文档**（如文章去重）。
  
  - 将文本转化为 64 位特征值，通过汉明距离判断相似度。

- **Semantic Vector (TKW.Extensions.Similarity.Vector)**：
  
  - 引入词向量（Word Embeddings）。
  
  - 解决“自行车”与“单车”这种字面完全不同但语义一致的问题。

### 6.2 性能监控建议

在未来的维护中，若遇到海量规则的批量查重需求，应考虑引入 **BK-Tree (Burkhard-Keller Tree)** 数据结构。这种结构利用三角不等式定理，可以在不遍历所有数据的情况下，快速从百万级词库中找出符合相似度阈值的候选集。

## 7. 维护规范

- **算法纯粹性**：`Common` 包内的算法严禁引入外部 NuGet 依赖。

- **性能回测**：任何针对算法逻辑的修改，必须通过 `BenchmarkDotNet` 进行内存和速度回测。

- **单元测试**：需覆盖“空字符串”、“完全相同”、“完全不同”、“长短不一”、“包含特殊字符”等边界 Case。
